# V2.6 Image Spike Report

Date: 2026-06-08

Scope: Part A only. Read-only investigation. No database writes, no schema changes, no imports, no source Excel modifications.

Script: `scripts/image-spike.ts`

## Executive Summary

Product image extraction is feasible for `.xlsx` files, but not through SheetJS worksheet objects. SheetJS 0.20.3 reads cell data and sheet structure, but it did not expose `worksheet["!images"]` in any tested file.

For `.xlsx`, images can be extracted by treating the workbook as a zip package:

1. Read image files from `xl/media/`.
2. Read worksheet drawing relationships from `xl/worksheets/_rels/sheetN.xml.rels`.
3. Read drawing anchors from `xl/drawings/drawingN.xml`.
4. Map `rId` values through `xl/drawings/_rels/drawingN.xml.rels`.
5. Use `xdr:from/xdr:row` and `xdr:from/xdr:col` to map images to sheet rows.

For `.xls`, image extraction is not reliable with the current project stack. SheetJS can read `.xls` cell data, but did not expose decoded images or usable image anchors. The local runtime also does not include LibreOffice, `xlrd`, `olefile`, or oletools.

Recommended Part B path: **Scenario 3 — `.xlsx` image extraction only; `.xls` continues as data-only import.**

Sources checked:
- [SheetJS Reading Files docs](https://docs.sheetjs.com/docs/api/parse-options) state the SheetJS Community Edition parse APIs focus on raw data and SheetJS Pro covers advanced features such as images.
- [SheetJS File Formats docs](https://docs.sheetjs.com/docs/miscellany/formats) confirm SheetJS reads `.xlsx` and `.xls`, which matches current import behavior.

## Detailed Results

### Case 1: Typical `.xlsx` — LED Bulbs

File: `LED Bulbs - Wellux  Price 202305.xlsx`

SheetJS result:
- Sheet names: `LED Bulb`
- `worksheet["!images"]`: not present
- image-related worksheet keys: none

Package result:
- `xl/media/`: 13 images
- media types: 12 JPEG, 1 PNG
- drawing anchors: 13
- anchors by sheet: `LED Bulb`: 13

Image examples:

| Media | Format | Dimensions | Bytes |
|---|---|---:|---:|
| `image1.jpeg` | JPEG | 238×237 | 3,553 |
| `image5.png` | PNG | 652×266 | 60,834 |
| `image6.jpeg` | JPEG | 4000×3000 | 3,567,894 |

Anchor examples:

| Media | Sheet row | Sheet col | Interpretation |
|---|---:|---:|---|
| `image3.jpeg` | 6 | B | Product row `C37 - 3W` |
| `image1.jpeg` | 10 | B | Product row `F37 - 3W` |
| `image2.jpeg` | 14 | B | Product row `G45-3W` |
| `image7.jpeg` | 19 | B | Product row `A60-7W` |
| `image5.png` | 1 | A | Company logo/header, should be ignored |

Reliability: **High** after excluding header/logo rows. The image row aligns with the product data row.

### Case 2: Legacy `.xls` — Waterproof Lighting Fixture

File: `Waterproof Lighting Fixture - Wellux 202305.xls`

SheetJS result:
- Sheet names: `WP-G`, `WP-H`, `WP-D`
- `worksheet["!images"]`: not present
- image-related worksheet keys: none
- `workbook.cfb.FullPaths`: present, 12 paths
- no obvious decoded image streams or image anchor records exposed

Additional local tool check:
- `openpyxl`: installed, but only supports `.xlsx`, not `.xls`
- `xlrd`: not installed
- `olefile` / oletools: not installed
- LibreOffice / `soffice`: not installed

Reliability: **Low / not usable for V2.6 integration**.

Practical conclusion: keep `.xls` import data-only for now. If `.xls` images become important, handle them with a separate external conversion pipeline later, likely LibreOffice `.xls → .xlsx` then reuse the `.xlsx` extraction path, or a dedicated BIFF/OLE parser.

### Case 3: Irregular `.xlsx` — Solar Floodlight / Streetlight

File: `核价 Welfull Wellux - Quotation- LED Solar Floodlight & Streetlight 20240516.xlsx`

SheetJS result:
- Sheet names: `Solar Wall Lamp`
- `worksheet["!images"]`: not present
- image-related worksheet keys: none

Package result:
- `xl/media/`: 16 images
- media types: 15 PNG, 1 JPEG
- drawing anchors: 48
- anchors by sheet: `Solar Wall Lamp`: 48
- anchor row range: row 1 to row 37

Important behavior:
- Multiple drawing anchors can point to the same media file.
- Example: `image1.png` appears at rows 7, 8, 9, 10, 11.
- Those rows are separate product variants (`SL-FH-50W`, `SL-FH-100W`, etc.) sharing the same product family image.

Reliability: **Medium**. Row anchors are available, but image-media mapping is not strictly one image per product. The importer should allow the same extracted image to attach to multiple product rows when the same media appears across repeated row anchors.

## Recommended Extraction Design

### Recommended Path

Use a project-local `.xlsx` image extractor:

```text
read xlsx zip
→ enumerate xl/media/*
→ parse worksheet rels
→ parse drawing anchors
→ map drawing rId → media file
→ map anchor row → import data row
→ store extracted image file
→ set products.image_path
```

Do not use SheetJS for the image objects. Continue using SheetJS for cell data and row parsing.

### Matching Rules

For `.xlsx` import:

1. Only consider anchors on the selected sheet.
2. Ignore anchors above the selected header row.
3. Prefer anchors whose `fromRowOneBased` equals the product row.
4. If a product row has multiple anchors, prefer image anchors in the expected image column if known.
5. If the same media appears on multiple rows, attach the same stored image path to each matching product.
6. Ignore obvious logo/header images:
   - rows before data start
   - very wide header image in row 1
   - anchors outside product table range

### Storage Recommendation

The schema already has `products.image_path`, so no new schema is required for a first implementation.

Recommended filesystem layout:

```text
data/images/products/{product_id}.{ext}
data/images/products/{product_id}-thumb.jpg
```

For duplicate/shared media:

```text
data/images/source/{source_file_id}/{sheet_name}/{media_name}
```

Then `products.image_path` can point to either:
- a product-specific copy, or
- a shared source image path reused by multiple products.

Recommended MVP choice: **copy or convert to product-specific image path** for simple UI rendering, while optionally preserving source images under `data/images/source/`.

### Image Quality / Compression

Preserve originals initially, but generate thumbnails for UI.

Reason:
- Some images are tiny enough to use directly.
- Some are huge, e.g. one bulb image is 4000×3000 and 3.6 MB.
- Product library pages should render thumbnails, not raw originals.

Suggested thumbnail size:
- Max width: 300 px
- Format: JPEG or WebP
- Keep original extension for source copy.

- source Excel files

## Recommendation for Part B

Proceed with **Scenario 3**:

1. `.xlsx` files: extract product images through zip + drawing anchors.
2. `.xls` files: keep data-only import.
3. Add image display to product library only after `.xlsx` extraction is stable.
4. Do not block batch import on image extraction failures. If image extraction fails for a file, import data and record a warning.

Reason:
- The `.xlsx` path is technically proven and does not require new heavy dependencies.
- The `.xls` path is not reliable with current tools.
- Most value comes from getting images for common `.xlsx` quote/catalog files without slowing the whole import pipeline.

## Open Questions Before Part B

1. Should product images be stored as source originals only, thumbnails only, or both?
2. For shared family images across variants, should all variants share the same image path?
3. Should image extraction run automatically for all `.xlsx` imports, or be an optional checkbox during import?
4. Should image failures be shown in import result warnings, similar to skipped rows?

Recommended defaults:
- Store both original and thumbnail.
- Allow shared family images.
- Default extraction ON for `.xlsx`, OFF/not available for `.xls`.
- Show warnings but never block data import.
