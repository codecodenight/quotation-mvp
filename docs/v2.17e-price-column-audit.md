## 根因

当前 `scripts/tube-bulb-split-apply.ts` / `scripts/tube-bulb-split-dryrun.ts` 的列检测逻辑继承自 V2.17C：

1. 每列只要 `parsePriceValue()` 能提取正数，就可能成为 price candidate。
2. 排序主要按 numeric count，未优先使用价格语义。
3. `No.` / `序号` 列、功率列、规格列里的数字非常密集，count 高。
4. 真正价格列有时含 `￥`，但行数较少或表头为空，排名反而落后。

因此错误选择了序号/规格列。

## 推荐修复策略

不要做逐条手工删除。推荐：

1. 恢复 V2.17D 前备份：
   - `backups/dev-before-v2.17d-tube-bulb-20260612-085146.sqlite`
2. 修正拆分导入脚本的价格列选择规则。
3. 重跑 V2.17C dry-run。
4. 人工 review 新 dry-run 报告里的 price column。
5. 再执行 apply。

理由：

- V2.17D 之后没有新的业务数据必须保留。
- 恢复备份可以一次性撤销：
  - 错误 new offers
  - 错误 updated offers
  - 错误 price_history
  - V2.17D 新建 files 记录
  - V2.17D 新建 products
- 比按 offer/price_history 做手术更可靠。

## 价格列选择规则建议

新规则：

1. 明确禁止以下列作为价格列，除非表头同时包含价格关键词：
   - `No.`
   - `NO.`
   - `序号`
   - `Item No.`
   - `产品名称`
   - `产品规格`
   - `W数`
   - `功率`
   - `电流`
   - `尺寸`
2. 价格列评分应优先看语义：
   - 表头含 `价格` / `单价` / `RMB` / `￥` / `含税` / `不含税` / `出厂价`
   - 单元格含 `￥` / `¥`
3. 对没有价格关键词、但能解析出数字的列，不应自动作为 price column。
4. dry-run 报告中如果 price column 无价格关键词，应标红/列入风险项。
