# Release Readiness Checklist

Last updated: 2026-06-26

这份清单定义本项目里的“可以发布”到底是什么意思。当前目标是
**Beta 私有内测版**，不是公开 SaaS，不是多用户平台，也不是完全交给非技术人员自行维护的正式产品。

## 发布定位

| 层级 | 含义 | 当前状态 |
|---|---|---|
| 内部日常可用 | 开发者/操作者能完整跑通报价流程，并能自己处理问题 | 已由 V2.0+ 达成 |
| Beta 私有内测 | 受过培训的用户能在私有服务器上使用；备份、恢复、支持边界有文档 | 当前目标 |
| 私有稳定版 | 非技术业务用户基本能独立操作，遇到常见问题知道怎么恢复 | 尚未达到 |
| 公开 SaaS | 多用户、账号体系、租户隔离、线上支持与安全合规 | 不在本项目范围 |

在所有 P0 项都有证据之前，不应把当前项目叫“正式发布版”。

## 评分规则

每个检查项按 0-3 分打分。

| 分数 | 含义 |
|---:|---|
| 0 | 未实现，或未验证 |
| 1 | 需要开发者协助，或需要直接改数据库/跑脚本 |
| 2 | 受过培训的操作者可以按文档重复完成 |
| 3 | 非技术用户也能理解操作与恢复步骤 |

发布门槛：

- **Beta 私有内测：** 所有 P0 项 >= 2 分；没有报价准确性阻塞项；至少完成 1 次备份恢复演练。
- **私有稳定版：** 所有 P0 项 = 3 分；所有 P1 项 >= 2 分；用户手册和恢复手册完整。
- **公开/SaaS 发布：** 不在此清单评估范围内，因为与项目约束冲突。

## 当前基线

以下只是基于现有文档的初始判断，不是发布认证。

| 维度 | 基线分 | 判断 |
|---|---:|---|
| 数据完整度 | 2 | 产品库、图片、参数、价格历史已有规模，但图片、CTN、源文件可用性和个别价格确认仍有缺口。 |
| 报价准确性 | 2 | 价格公式、预览、健康警告、导出测试已有；还缺固定的 golden sample 验收包。 |
| 用户操作流 | 1 | 核心流程能跑，但非技术用户仍缺登录、导入、修数据、错误处理说明。 |
| 错误处理 | 1 | 报价警告较强；导入、文件、备份、部署失败还缺用户可理解的恢复说明。 |
| 报价审计 | 2 | 已有价格历史和源文件链路；还缺每张报价的标准审计检查动作。 |
| 备份恢复 | 1 | V43 已有备份脚本；尚未记录恢复演练证据。 |
| 部署运维 | 1 | PM2/nginx/deploy 脚本已有；首次部署、回滚、服务器验证还需要实测。 |
| 安全权限 | 1 | Basic Auth 只适合私有内测；HTTPS、防火墙、密钥和暴露面仍需确认。 |

| ID | 优先级 | 检查项 | 通过证据 | 分数 |
|---|---|---|---|---:|
| DATA-01 | P0 | 日常报价高频品类能搜到产品、有效报价和可用款号。 | 品类覆盖报告 + 样例搜索截图。 |  |
| DATA-02 | P0 | Beta 报价涉及的产品有足够客户可见信息，导出不出现中文内控/包装垃圾文本。 | 样例报价预览中无客户可见 P0 警告。 |  |
| DATA-03 | P0 | CTN/MOQ/Size 缺失会在导出前可见，操作者能判断是否接受。 | `/data-quality` 快照 + 报价警告样例。 |  |
| DATA-04 | P1 | 常用销售场景有产品图；无图时用户能明确知道。 | 按品类的图片覆盖率报告。 |  |
| DATA-05 | P1 | 已知脏数据组都有处理决定：修复、排除、或作为限制接受。 | 清洗/审计报告链接。 |  |

| QUOTE-04 | P0 | 报价预览与实际导出的 Excel 逐行一致。 | 预览/导出验收报告。 |  |
| QUOTE-05 | P1 | 历史客户 FOB 售价与供应商采购价清楚分离。 | 历史报价参考样例。 |  |

## Beta 验收脚本

叫 Beta-ready 之前，用一名受训用户跑一次：

1. 从无登录态浏览器访问并登录。
2. 搜索 3 个常用产品，覆盖至少 2 个品类。
3. 选择供应商报价，并比较推荐报价与备选报价。
4. 创建 1 张同币种报价和 1 张跨币种报价。
5. 查看报价警告，并从 UI 修复 1 个产品资料问题。
6. 导出客户模式 Excel，人工复算 5 行价格。
7. 从报价历史重新打开该报价，确认快照值。
8. 导入 1 份正常供应商 Excel，并确认导入结果。
9. 执行 1 次备份，将备份恢复到临时副本，并从恢复后的 DB 启动应用。
10. 记录所有需要开发者介入的节点。

通过条件：用户完成 1-9 步且不需要直接改数据库；第 10 步里记录的问题要么补进文档，要么升级为 P0/P1 检查项。

## No-Go 条件

出现任一情况，不进入 Beta：

- 没有成功的备份恢复演练。
- 报价预览和导出 Excel 在样例报价中不一致。
- 普通用户无法判断警告是客户可见问题，还是物流/内部提醒。
- 服务器外部可访问但没有认证。
- 源数据或导出报价文件存在被应用覆盖的风险。
- 正常报价流程仍需要手工改 SQLite。
